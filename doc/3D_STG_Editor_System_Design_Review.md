# 3D STG 编辑系统 — 策划逻辑梳理与建议

> 本文档基于用户提出的编辑系统设计方案，进行逻辑检查、结构梳理，并提供细化方向与建议。
> 标注规则：`[🤖建议]` 标记的内容为 AI 的补充建议，非用户原始设计。

---

## 一、项目全局定位

### 1.1 三种游戏模式定义

| 模式 | 视角 | 移动方式 | 场景边界 | 镜头行为 |
|------|------|----------|----------|----------|
| A. 传统2D STG | 固定正交/近正交 | 平面四向/八向 | 固定矩形 | 固定或缓慢跟随 |
| B. 3D自由移动STG | 自由第三人称/第一人称 | 全向六自由度 | 3D包围体(球/盒) | 玩家控制或半自动 |
| C. 3D脚本演出STG | 第三人称受限 | 受脚本约束的移动 | 动态变化(脚本驱动) | 完全由脚本控制 |

### 1.2 模式切换的设计意图

- 模式切换由关卡作者设计，非玩家选择
- 后两种3D模式占比较大
- 切换不应有割裂感

### 1.3 逻辑检查：模式切换的无割裂感

要实现"无割裂感"的模式切换，需要明确以下约束：

**必须统一的要素：**
- 玩家角色的视觉表现在三种模式间需要连续（不能突然变形/消失）
- BGM 和音效不能因切换中断
- 弹幕在切换瞬间的存续规则（已发射的弹幕是消失、转化、还是保留？）
- 分数/资源/状态的连续性

**需要过渡动画的要素：**
- 镜头从自由控制到脚本控制的平滑插值
- 场景边界从3D包围体到2D矩形的视觉变形
- 玩家移动自由度的渐变约束（不能一帧内突然锁轴）

> `[🤖建议]` 建议在数据模型中为"模式切换"定义一个独立的 **Transition 事件类型**，包含：
> - 源模式 / 目标模式
> - 过渡时长
> - 镜头插值曲线
> - 边界变形曲线
> - 弹幕存续策略（清除 / 转化 / 冻结）
> - 玩家控制权渐变参数
>
> 这样编辑器中可以把"模式切换"作为时间轴上的一个可编辑事件，而不是硬编码行为。

### 1.4 待细化：模式B与模式C的边界

用户描述中，模式B（自由移动）和模式C（脚本演出）的区别主要在于：
- B：玩家完全控制移动和视角
- C：移动和视角受脚本约束，场景边界动态变化

但实际游戏中可能存在**混合态**——比如脚本控制镜头但玩家仍可自由移动。

> `[🤖建议]` 建议不要把模式定义为离散的三种，而是定义为一组**连续可调的参数轴**：
> - 移动自由度：0（完全锁定）→ 1（完全自由）
> - 镜头控制权：0（完全脚本）→ 1（完全玩家）
> - 边界类型：静态 / 动态脚本驱动
> - 视角投影：正交 ⟷ 透视（可插值）
>
> 三种"模式"只是这些参数的预设组合。这样模式切换就自然变成参数的渐变，天然无割裂。

## 二、编辑系统总体架构

### 2.1 系统分层

```
┌─────────────────────────────────────────────┐
│            互动式编辑器 (Unity-based)          │
├─────────────────────────────────────────────┤
│  工具层：场景工具 | 演出脚本 | 弹幕/符卡编辑    │
├─────────────────────────────────────────────┤
│  时间轴系统 (全局 + 子时间轴)                   │
├─────────────────────────────────────────────┤
│  存储层：运行时二进制 ⟷ YAML/TOML 人类可读     │
├─────────────────────────────────────────────┤
│  数据模型层：关卡 > 符卡 > 小怪波次 > 弹幕模式   │
└─────────────────────────────────────────────┘
```

### 2.2 数据流向

```
用户操作(编辑器UI)
    │
    ▼
工具层(场景/脚本/弹幕编辑器)
    │
    ▼
数据模型(内存中的关卡数据结构)
    │
    ├──▶ 时间轴系统(可视化 + 回放预览)
    │
    ├──▶ 存储层 ──▶ YAML(编辑用) / Binary(运行用)
    │
    └──▶ 运行时预览(在编辑器内直接试玩)
```

> `[🤖建议]` 编辑器应该有一个明确的"数据模型层"作为 Single Source of Truth。
> 所有工具组都读写同一个数据模型，时间轴系统只是这个模型的一种可视化投影。
> 这样可以避免"时间轴数据"和"工具数据"不同步的问题。

## 三、存储系统

### 3.1 双格式设计

| 格式 | 用途 | 特点 |
|------|------|------|
| 运行时格式 | 游戏实际加载 | 二进制/FlatBuffers/MessagePack，IO友好，体积小 |
| 编辑格式 | 第三方编辑/版本控制 | YAML 或 TOML，人类可读，diff友好 |

双向转化流程：
```
编辑格式(YAML) ──序列化──▶ 运行时格式(Binary)
运行时格式(Binary) ──反序列化──▶ 编辑格式(YAML)
```

### 3.2 逻辑检查：双格式的一致性保证

**问题：谁是权威源？**

双格式互转听起来简单，但必须回答一个核心问题：当两种格式的文件同时存在且内容冲突时，以谁为准？

可能的策略：
- **编辑器内操作 → 只修改内存数据模型 → 保存时同时输出两种格式**（推荐）
- 手动编辑 YAML 后导入 → 覆盖内存模型 → 重新导出二进制
- 二进制永远由 YAML 生成，不允许单独修改二进制

> `[🤖建议]` 推荐采用 **YAML 为权威源** 的策略：
> - 编辑器保存时输出 YAML
> - 构建/打包时由 YAML 编译为二进制（类似 protobuf 的 .proto → .bin 流程）
> - 二进制格式不允许反向修改，只允许从 YAML 单向生成
> - 这样版本控制只需要跟踪 YAML 文件，二进制文件可以 .gitignore
>
> 如果确实需要"从二进制反序列化回 YAML"的能力（比如从别人的成品关卡中提取数据），
> 可以作为一个独立的"导入/逆向"工具存在，但不应该是日常工作流的一部分。

### 3.3 待细化：格式选型

| 候选 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| YAML | 人类可读性最好，支持注释，层级清晰 | 缩进敏感，大文件性能差 | 弹幕模式、符卡定义等结构化数据 |
| TOML | 简洁，适合配置 | 不擅长深层嵌套 | 全局设置、关卡元数据 |
| JSON | 通用性最强 | 无注释，冗长 | 与外部工具交互 |

> `[🤖建议]` 考虑到弹幕数据的嵌套深度（关卡 > 波次 > 弹幕组 > 单弹幕参数），
> YAML 是更合适的选择。TOML 可以用于顶层配置（项目设置、难度参数表等）。
> 建议不要混用太多格式，主体用 YAML，配置用 TOML，最多两种。

### 3.4 待细化：版本兼容性

编辑格式需要考虑版本迁移：
- 数据结构会随开发演进
- 旧版本保存的 YAML 需要能被新版本编辑器加载

> `[🤖建议]` 在 YAML 文件头部加入 `schema_version` 字段，
> 编辑器加载时根据版本号执行迁移脚本（migration）。
> 这是所有长期维护的编辑器都需要的基础设施，越早建立越好。

## 四、时间轴系统

### 4.1 用户原始设计梳理

用户描述的时间轴系统包含两类内容：

**A. 道中小怪时间轴（精确时间）**
- 小怪出现的时机需要精确控制
- 时间误差要求小

**B. Boss 符卡间隔时间轴（弹性时间）**
- 符卡击破时间由符卡类本身决定（玩家表现相关，不固定）
- 符卡之间的间隔是设计者控制的（用于放松节奏 + BGM 配合）
- 时间轴需要能反映符卡本身的时长

### 4.2 逻辑检查：精确时间 vs 弹性时间的矛盾

这里存在一个**核心设计矛盾**：

道中小怪的时间是绝对的（第30秒出现就是第30秒），但 Boss 符卡的时间是相对的（上一张符卡被击破后 N 秒开始下一张）。如果道中和 Boss 战在同一条时间轴上，那么：

- Boss 战开始时间取决于道中部分玩家花了多久
- 如果道中有"不击破就一直存在"的敌人，时间轴就不是线性的

**这意味着时间轴不能是一条简单的绝对时间线。**

### 4.3 建议的时间轴模型

> `[🤖建议]` 建议采用 **分段式时间轴 + 事件触发** 的混合模型：
>
> ```
> 全局时间轴（关卡级）
> ├── 段落1: 道中A [绝对时间轴, 0:00 ~ 1:30]
> │   ├── t=0:05  小怪波次1
> │   ├── t=0:20  小怪波次2
> │   └── t=1:25  中Boss登场（触发条件：时间到达 OR 前置波次全灭）
> │
> ├── 段落2: 中Boss战 [事件驱动时间轴]
> │   ├── 符卡1 [时长上限: 60s, 击破后 → 间隔]
> │   ├── 间隔1 [固定3s, 用于BGM过渡]
> │   ├── 符卡2 [时长上限: 90s, 击破后 → 间隔]
> │   └── 击破/超时 → 触发段落3
> │
> ├── 段落3: 道中B [绝对时间轴, 0:00 ~ 2:00]
> │   └── ...
> │
> └── 段落4: 最终Boss战 [事件驱动时间轴]
>     └── ...
> ```
>
> 关键设计点：
> - 每个"段落"内部有自己的本地时间轴（从0开始）
> - 段落之间通过**触发条件**连接，而非绝对时间
> - 道中段落使用绝对时间（精确控制小怪出现）
> - Boss段落使用事件驱动时间（符卡击破 → 间隔 → 下一符卡）
> - 全局时间轴只负责段落的排列顺序和触发条件

### 4.4 符卡时间的双重性质

用户提到"符卡击破时间由符卡类本身决定"，这需要进一步拆解：

| 属性 | 由谁决定 | 说明 |
|------|----------|------|
| 符卡时间上限 | 设计者 | 超时则强制结束（miss/bomb 扣分等） |
| 符卡实际耗时 | 玩家表现 | 0 ~ 时间上限之间 |
| 符卡间隔时长 | 设计者 | 固定值，用于节奏控制和BGM配合 |
| 符卡间隔内的演出 | 设计者 | 可以放对话、镜头运动、场景变化等 |

> `[🤖建议]` 符卡间隔不应该只是"空白等待"，而应该是一个可编辑的**演出片段**。
> 在编辑器中，符卡间隔应该和符卡本身一样，拥有自己的子时间轴，
> 可以放置镜头脚本、对话事件、场景变化等。
> 这样 BGM 配合就不只是"等 N 秒"，而是"在这 N 秒内执行一段精心设计的演出"。

### 4.5 时间轴的可视化需求

用户提到"时间轴本身应该通过不同的设计元素在自定义的互动式编辑器可视化"。

需要可视化的元素：

| 元素 | 可视化方式 | 交互方式 |
|------|-----------|----------|
| 道中小怪波次 | 时间轴上的色块/标记 | 拖拽调整时间点 |
| Boss 符卡 | 可伸缩的时间块（反映时间上限） | 拖拽调整顺序和间隔 |
| 符卡间隔 | 符卡块之间的间隙区域 | 拖拽调整间隔时长 |
| 模式切换事件 | 特殊标记/分割线 | 点击编辑切换参数 |
| BGM 波形 | 背景层波形图 | 只读参考，用于对齐节奏 |
| 段落边界 | 垂直分割线 + 触发条件标签 | 点击编辑触发条件 |

> `[🤖建议]` 强烈建议在时间轴编辑器中叠加 **BGM 波形显示**。
> 既然符卡间隔的一个重要目的是配合 BGM 进行情绪调动，
> 那么设计者需要能直观地看到"这个间隔对应 BGM 的哪个段落"。
> 类似视频编辑软件中音轨和视频轨的对齐方式。

### 4.6 子时间轴的递归结构

用户提到"每个设计都应该包括子时间轴"。这意味着时间轴是递归的：

```
关卡时间轴
├── 段落时间轴（道中/Boss战）
│   ├── 波次/符卡时间轴
│   │   ├── 单个小怪行为时间轴（移动路径、射击时机）
│   │   └── 弹幕模式时间轴（发射节奏、变化曲线）
│   └── 演出事件时间轴（镜头、对话、场景变化）
```

> `[🤖建议]` 递归时间轴的 UI 设计是一个重大挑战。建议：
> - 采用"聚焦 + 面包屑"模式：编辑器同一时刻只显示一层时间轴的细节
> - 上层时间轴中的元素可以双击"钻入"查看子时间轴
> - 顶部显示面包屑导航：`关卡 > 段落2:Boss战 > 符卡3 > 弹幕模式A`
> - 避免试图在一个视图中同时显示所有层级——信息密度会爆炸

## 五、互动式编辑工具包

### 5.1 工具组总览

| 工具组 | 职责 | 依赖 |
|--------|------|------|
| 1. 场景模型调整 | 场景中3D模型的摆放、材质、光照 | Unity Scene View |
| 2. 场景演出脚本 | 镜头、边界、场景运动、模式切换 | 时间轴系统 |
| 3. 从小到大的设计 | 弹幕→小怪→符卡→整关 | 时间轴系统 + 存储系统 |
| 4. 2D⟷3D转化 | 同一设计在不同模式间的等价转换 | 工具组3的输出 |

**工具组之间的依赖关系：**
```
工具组1(场景) ──提供场景上下文──▶ 工具组2(演出脚本)
                                      │
工具组3(弹幕/符卡设计) ◀──时间轴驱动──┘
         │
         ▼
工具组4(2D⟷3D转化) ◀── 消费工具组3的设计数据
```

### 5.2 工具组1：场景设计用模型调整工具组

用户描述："可与 Unity 本身搭配的、场景设计用的模型调整工具组"

**梳理：这个工具组的边界在哪里？**

Unity 本身已经有强大的场景编辑能力（Transform、Prefab、材质编辑等）。这个工具组需要明确：它是 Unity 原生功能的**补充**，还是**替代**？

合理的定位应该是：在 Unity Scene View 之上提供**STG 特化的辅助功能**，例如：

| 功能 | 说明 | Unity原生是否已有 |
|------|------|-------------------|
| 场景边界可视化 | 显示当前模式下的活动边界 | 无，需自定义 Gizmo |
| 弹幕发射点标记 | 在场景中标记 Boss/小怪的弹幕发射位置 | 无，需自定义 |
| 模式预览切换 | 一键切换查看同一场景在2D/3D模式下的样子 | 无，需自定义 |
| 安全区域标记 | 标记玩家可达区域 vs 危险区域 | 无，需自定义 |
| 路径编辑器 | 小怪/Boss 的移动路径可视化编辑 | 部分（需扩展） |

> `[🤖建议]` 这个工具组不应该试图替代 Unity 的场景编辑器，
> 而应该以 **Custom Editor Window + Scene View Overlay + Gizmo** 的形式存在。
> 用户仍然在 Unity Scene View 中工作，但看到的是增强后的视图。
> 
> 具体实现建议：
> - 使用 `EditorWindow` 做工具面板
> - 使用 `Handles` API 在 Scene View 中绘制边界、路径等
> - 使用 `SceneView.duringSceneGui` 注入自定义交互
> - 使用 `CustomEditor` 为关卡数据对象提供 Inspector 面板

### 5.3 工具组2：场景演出脚本工具组

用户描述的脚本类型：
1. 镜头控制脚本
2. 场景（活动）边界控制脚本
3. 场景运动脚本
4. 模式切换脚本

### 逻辑检查：脚本的执行模型

**问题：这些"脚本"是什么？**

用户说的"脚本"有两种可能的理解：
- **A. 数据驱动的指令序列**（关键帧 + 插值曲线，类似动画）
- **B. 可编程的代码脚本**（类似 Lua/VisualScript，图灵完备）

这两种方案的复杂度差异巨大。

| 方案 | 优点 | 缺点 | 复杂度 |
|------|------|------|--------|
| A. 数据驱动 | 简单、可预测、易序列化 | 表达力有限 | 低 |
| B. 可编程脚本 | 表达力强、灵活 | 调试困难、安全风险、序列化复杂 | 高 |
| C. 混合方案 | 兼顾两者 | 需要明确边界 | 中 |

> `[🤖建议]` 推荐 **方案C：混合方案**，具体为：
> - 镜头控制：数据驱动（关键帧 + 曲线），覆盖 90% 的需求
> - 边界控制：数据驱动（形状参数随时间变化的曲线）
> - 场景运动：数据驱动（路径 + 速度曲线）
> - 模式切换：数据驱动（Transition 事件，参见 1.3 节）
> - 特殊需求：提供简单的条件/触发器系统（if-then 规则，非图灵完备）
>
> 只有当数据驱动方案确实无法满足需求时，才考虑引入脚本语言。
> 过早引入脚本系统会极大增加编辑器的复杂度。

### 各脚本类型的数据结构草案

**镜头控制脚本：**
```yaml
camera_script:
  type: "cinematic"
  keyframes:
    - time: 0.0
      position: [0, 5, -10]
      look_at: [0, 0, 0]
      fov: 60
      easing: "ease_in_out"
    - time: 2.0
      position: [5, 3, -8]
      look_at: [0, 1, 0]
      fov: 45
      easing: "linear"
  blend_in: 0.5   # 从玩家控制过渡到脚本控制的时间
  blend_out: 0.3  # 从脚本控制过渡回玩家控制的时间
```

**边界控制脚本：**
```yaml
boundary_script:
  keyframes:
    - time: 0.0
      shape: "sphere"
      params: { radius: 50 }
    - time: 3.0
      shape: "box"
      params: { width: 40, height: 30, depth: 40 }
      easing: "ease_out"
  # 边界形状在 sphere 和 box 之间插值变形
```

**模式切换脚本：**
```yaml
mode_transition:
  trigger: "boss_phase_2_start"
  from_preset: "3d_free"
  to_preset: "3d_scripted"
  duration: 1.5
  params:
    movement_freedom: { curve: "ease_in_out", from: 1.0, to: 0.3 }
    camera_control: { curve: "ease_in", from: 1.0, to: 0.0 }
    projection: { from: "perspective", to: "perspective" }  # 不变

### 逻辑检查："从小到大"的编辑流程

用户强调"让用户从最小的地方开始做"。这个设计理念是正确的，但需要注意：

**问题1：弹幕模式脱离上下文是否有意义？**

一个弹幕模式（BulletPattern）在脱离符卡/小怪上下文时，缺少以下信息：
- 发射者的位置和移动方式
- 场景边界（影响弹幕的有效范围）
- 当前处于哪种游戏模式（2D/3D）

> `[🤖建议]` 弹幕模式编辑器应该提供一个**沙盒预览环境**：
> - 默认提供一个静止的发射点 + 标准边界
> - 用户可以选择"在XX符卡的上下文中预览"来获得真实环境
> - 这样既能独立编辑，又能在上下文中验证

**问题2：小怪设计的复用性**

小怪设计应该分为两层：
- **小怪类型**（EnemyType）：外观、血量、基础行为——可复用的模板
- **小怪实例**（EnemyInstance）：具体出现在哪个波次、什么位置、什么路径——关卡特定的

> `[🤖建议]` 编辑器应该有一个**资源库面板**，存放所有已定义的：
> - 弹幕模式模板
> - 小怪类型模板
> - 符卡模板
> - 演出脚本模板
>
> 用户在设计关卡时，从资源库中拖拽模板到时间轴上，然后在实例上做个性化调整。
> 这类似于 Unity 的 Prefab 机制——模板是 Prefab，实例是场景中的 GameObject。

**问题3：符卡设计的完整性**

一个符卡（SpellCard）需要包含的内容远不止弹幕模式：

| 组成部分 | 说明 |
|----------|------|
| 弹幕模式 | 一张符卡可能有多个弹幕模式，按阶段切换 |
| Boss 移动模式 | Boss 在这张符卡期间的移动行为 |
| 血量/时间上限 | 决定符卡何时结束 |
| 阶段切换条件 | 血量阈值触发弹幕模式变化 |
| 视觉效果 | 符卡宣言、背景变化、特效 |
| 音效/BGM变化 | 符卡开始时的音效、BGM切换 |
| 得分规则 | 击破奖励、时间奖励等 |
| 模式相关参数 | 这张符卡在2D/3D模式下的差异化参数 |

### 5.5 工具组4：2D⟷3D 模式转化类

用户描述的核心例子：
> "让一个3D场景中由球体表面均匀向外发射弹幕的设计在2D场景中的行为变为由圆形向外均匀发射，并且实现难度近似的弹幕密度转化"

### 逻辑检查：这是整个系统中最复杂的部分

**问题1：转化的数学基础**

3D球面均匀分布 → 2D圆形均匀分布，看似简单，但涉及：
- 球面上 N 个点的均匀分布（Fibonacci sphere 等算法）投影到圆上后不再均匀
- "难度近似"如何量化？弹幕密度在3D中是 弹/立体角，在2D中是 弹/弧度
- 玩家的闪避空间在3D中是体积，在2D中是面积——维度不同，难度感知不同

**问题2："1对1关系"的局限性**

用户说"建立1对1关系的对应转化类"，但并非所有3D弹幕设计都有自然的2D对应：

| 3D 设计 | 2D 对应 | 转化难度 |
|---------|---------|----------|
| 球面均匀发射 | 圆形均匀发射 | 低（投影） |
| 螺旋上升弹幕 | 螺旋扩散弹幕 | 中（降维） |
| 利用Z轴深度的弹幕层 | ？ | 高（信息丢失） |
| 需要玩家上下移动躲避的弹幕 | ？ | 高（自由度丢失） |
| 3D空间中的弹幕墙（有洞） | 2D弹幕墙（有洞） | 中（截面投影） |

**问题3：转化的方向性**

用户似乎假设设计者先做3D版本，再转化为2D版本。但实际工作流可能是：
- 有些弹幕先在2D中设计（传统STG经验），再扩展到3D
- 有些弹幕只在3D中有意义，2D中需要完全不同的替代设计

> `[🤖建议]` 建议将"转化"重新定义为三个层次：
>
> **层次1：自动转化（参数映射）**
> - 适用于简单几何关系的弹幕（球面→圆形、柱面→线性等）
> - 编辑器提供预定义的转化规则
> - 密度转化公式：`density_2d = density_3d * (4π / 2π) * difficulty_factor`
>   （其中 difficulty_factor 是可调参数，用于手动微调难度感知）
>
> **层次2：辅助转化（模板 + 手动调整）**
> - 适用于有对应关系但需要手动调整的弹幕
> - 编辑器生成一个"初始转化版本"，设计者在此基础上修改
> - 提供 A/B 对比预览（左边3D，右边2D）
>
> **层次3：独立设计（仅标记对应关系）**
> - 适用于无法自动转化的弹幕
> - 设计者分别为2D和3D创建独立的弹幕设计
> - 编辑器只负责标记"这两个设计是同一个逻辑符卡的不同模式版本"
> - 确保它们在关卡流程中的位置一致
>
> 这三个层次不是互斥的——同一个符卡中的不同弹幕模式可以使用不同层次的转化策略。

### 难度等价性的量化

> `[🤖建议]` "难度近似"需要一个可量化的指标体系，否则无法自动化。建议定义：
>
> - **弹幕密度**：单位时间内通过玩家判定点附近单位面积/体积的弹幕数
> - **安全区域比例**：玩家可安全站立的区域占总活动区域的比例
> - **反应时间**：从弹幕进入可见范围到到达玩家位置的最短时间
> - **路径复杂度**：安全路径的最小转向次数/角度变化
>
> 转化时，编辑器可以计算这些指标在2D和3D版本中的值，
> 并高亮差异过大的部分，提示设计者手动调整。
> 这不需要完美——只需要给设计者一个参考基准。

## 六、逻辑问题汇总与分析

### 问题1（严重）：时间轴的时间模型未定义

**现状：** 用户描述中混合了绝对时间（道中小怪）和相对时间（符卡间隔），但没有明确时间轴的时间模型。

**影响：** 如果不解决这个问题，时间轴编辑器的核心数据结构无法确定。一条绝对时间轴无法正确表示"符卡击破后等待3秒"这种事件驱动的行为。

**建议方案：** 见第四章 4.3 节的分段式时间轴模型。

---

### 问题2（严重）：2D⟷3D 转化的可行性边界未定义

**现状：** 用户期望建立"1对1"的转化关系，但很多3D弹幕设计在降维到2D时会丢失信息，无法自动转化。

**影响：** 如果按"所有弹幕都必须有自动转化"来设计系统，会导致：
- 3D弹幕设计被限制在"能转化为2D"的子集内，创意受限
- 或者转化系统变得极其复杂，开发成本失控

**建议方案：** 见第五章 5.5 节的三层次转化策略。

---

### 问题3（中等）：编辑器与 Unity 的职责边界模糊

**现状：** 用户说"基于 Unity"的互动式编辑器，但没有明确哪些功能用 Unity 原生能力，哪些需要自定义。

**影响：** 可能导致：
- 重复造轮子（重新实现 Unity 已有的功能）
- 或者过度依赖 Unity Editor API（导致编辑器只能在 Unity Editor 中运行，无法独立分发）

**建议方案：**
- 场景编辑：利用 Unity Scene View + 自定义 Gizmo/Handles
- 时间轴编辑：自定义 EditorWindow（Unity 的 Timeline 包不适合 STG 的需求）
- 弹幕预览：利用 Unity 的 Play Mode / EditMode 测试
- 数据编辑：自定义 Inspector + EditorWindow

---

### 问题4（中等）：子时间轴的递归深度与 UI 复杂度

**现状：** 用户要求每个设计项都有子时间轴，形成递归结构。

**影响：** 递归时间轴的 UI 设计非常困难。如果处理不好，用户会迷失在层级中。

**建议方案：** 见第四章 4.6 节的"聚焦 + 面包屑"模式。

---

### 问题5（低）：存储系统的双格式维护成本

**现状：** 双格式互转需要维护两套序列化/反序列化代码。

**影响：** 每次数据结构变更都需要同时更新两种格式的处理代码，维护成本翻倍。

**建议方案：** 见第三章 3.2 节，采用 YAML 为权威源的单向生成策略。

---

### 问题6（需确认）：三种模式是否共享同一套弹幕系统

**现状：** 用户描述了三种游戏模式，但没有明确弹幕系统在三种模式下是否是同一套代码。

**需要回答的问题：**
- 2D 模式的弹幕是 3D 弹幕的正交投影，还是完全独立的 2D 弹幕系统？
- 如果是投影，那么 2D 模式下弹幕仍然在 3D 空间中运动，只是相机是正交的？
- 如果是独立系统，那么转化类需要在两套完全不同的弹幕系统之间建立映射？

> `[🤖建议]` 推荐采用 **统一3D弹幕系统 + 模式约束层** 的架构：
> - 所有弹幕始终在3D空间中运动
> - 2D模式 = 正交相机 + 将弹幕和玩家的Z轴运动锁定在一个平面上
> - 3D自由模式 = 透视相机 + 无约束
> - 3D脚本模式 = 透视相机 + 脚本约束
>
> 这样弹幕系统只需要一套，模式切换只是改变约束条件和相机参数。
> 转化类的工作也简化为：调整弹幕参数使其在约束条件下的难度等价。

## 九、缺漏设计分析

> `[🤖建议]` 以下是当前方案中完全没有提及、但对编辑系统的可用性和完整性至关重要的缺漏。

### 缺漏1（关键）：撤销/重做系统（Undo/Redo）

当前设计完全没有提及编辑操作的可逆性。对于一个互动式编辑器来说，这不是"锦上添花"，而是**基础设施**。

没有 Undo 的编辑器，用户每一步操作都是不可逆的赌博——尤其是在时间轴上拖拽元素、调整弹幕参数这种高频操作中，误操作是必然的。

需要覆盖的操作类型：
- 时间轴上的元素移动/删除/添加
- 弹幕参数修改
- 场景中的模型摆放
- 演出脚本的关键帧编辑
- 符卡/小怪的创建和删除

Unity 自带 `Undo.RecordObject` / `Undo.RegisterCompleteObjectUndo` 等 API，但它们只对 UnityEngine.Object 有效。如果数据模型是纯 C# 类（ScriptableObject 之外的），需要自己实现 Command Pattern 的撤销栈。

> `[🤖建议]` 这个问题应该在 Phase 0 就决定数据模型的宿主形式：
> - 如果数据模型全部基于 ScriptableObject → 可以直接用 Unity Undo 系统
> - 如果数据模型是纯 POCO/纯 C# 类 → 需要自建 Command Pattern 撤销栈
> - 混合方案：编辑器内用 ScriptableObject 承载（享受 Unity Undo），保存时序列化为 YAML
>
> 推荐混合方案。这也直接影响存储系统的设计。

### 缺漏2（关键）：玩家角色系统的缺失

整个设计方案围绕"弹幕"、"小怪"、"Boss"、"场景"展开，但完全没有提到**玩家角色**在编辑器中的表示。

编辑器需要回答：
- 玩家的判定点大小是多少？（直接影响弹幕难度感知）
- 玩家的移动速度在三种模式下分别是多少？（影响安全路径计算）
- 玩家有哪些能力？（射击、bomb、减速移动、闪避？）
- 玩家能力如何影响符卡设计？（比如 bomb 会清屏，那符卡的弹幕密度设计需要考虑 bomb 的存在）
- 编辑器中预览弹幕时，是否需要一个"虚拟玩家"来模拟闪避？

> `[🤖建议]` 在数据模型中需要一个 `PlayerProfile` 定义：
> ```yaml
> player_profile:
>   hitbox_radius: 0.02        # 判定点半径（归一化到场景尺度）
>   move_speed_2d: 4.5         # 2D模式移动速度
>   move_speed_3d: 6.0         # 3D模式移动速度
>   focus_speed_ratio: 0.4     # 低速移动倍率
>   bomb_clear_radius: 15.0    # bomb 清屏半径
>   graze_radius: 0.5          # 擦弹判定半径
> ```
> 弹幕预览时，编辑器应该能放置一个"测试用玩家"来直观感受难度。
> 难度指标的计算也依赖这些参数。

### 缺漏3（关键）：难度等级系统

你提到了"难度近似的弹幕密度转化"，但整个设计中没有提到游戏本身的**难度等级系统**（Easy/Normal/Hard/Lunatic 等）。

这对编辑器的影响是根本性的：
- 每个弹幕模式是否需要为每个难度等级分别设计？
- 还是定义一个基础版本 + 难度缩放参数？
- 难度影响的是弹幕数量？速度？密度？还是完全不同的弹幕模式？

如果不在编辑器层面解决这个问题，设计者要么：
- 为每个难度等级重复做一遍所有设计（工作量爆炸）
- 或者只做一个难度，后续再手动调整（容易遗漏）

> `[🤖建议]` 建议采用 **基础设计 + 难度修饰器** 的模式：
> ```yaml
> bullet_pattern:
>   base:
>     count: 24
>     speed: 3.0
>     spread_angle: 360
>   difficulty_modifiers:
>     easy:   { count_mult: 0.5, speed_mult: 0.7 }
>     normal: { count_mult: 1.0, speed_mult: 1.0 }  # 基准
>     hard:   { count_mult: 1.5, speed_mult: 1.2 }
>     lunatic: { count_mult: 2.0, speed_mult: 1.5, extra_layers: 1 }
> ```
> 编辑器中可以一键切换难度预览，看到同一个弹幕在不同难度下的表现。
> 对于 Lunatic 等极端难度，允许 `extra_layers` 这种结构性变化，而不仅仅是数值缩放。

### 缺漏4（重要）：碰撞/判定系统的编辑器表示

弹幕游戏的核心体验取决于碰撞判定的精确性。但当前设计中，编辑器只关注弹幕的"视觉表现"和"发射参数"，没有提到：

- 弹幕的碰撞体形状（圆形？胶囊体？多边形？）
- 碰撞体与视觉表现的关系（通常碰撞体比视觉小）
- 擦弹（graze）判定区域
- 编辑器中碰撞体的可视化

> `[🤖建议]` 弹幕编辑器中应该有一个"碰撞体叠加显示"开关：
> - 显示每颗弹幕的实际碰撞体（红色线框）
> - 显示擦弹判定区域（黄色线框）
> - 显示玩家判定点（绿色点）
> - 这对于调试"为什么这个弹幕看起来没碰到但判定被弹了"至关重要

### 缺漏5（重要）：预览/回放系统

设计中提到了"互动式编辑器"，但没有明确编辑器内的预览能力边界：

| 预览需求 | 复杂度 | 说明 |
|----------|--------|------|
| 单弹幕模式预览 | 低 | 沙盒中播放一个弹幕模式 |
| 符卡完整预览 | 中 | 弹幕 + Boss移动 + 阶段切换 |
| 道中段落预览 | 中 | 多波次小怪 + 时间轴驱动 |
| 整关预览 | 高 | 需要完整的游戏循环 |
| 带玩家操控的预览 | 高 | 需要玩家控制器 + 碰撞检测 |
| 倒带/帧步进 | 高 | 需要确定性回放或状态快照 |

> `[🤖建议]` 预览系统应该分层实现：
> - Phase 2 就实现"单弹幕模式的纯视觉预览"（不需要碰撞，只看弹幕轨迹）
> - Phase 4 实现"符卡预览"（加入 Boss 移动）
> - Phase 6 实现"带虚拟玩家的可操控预览"
> - 倒带/帧步进作为高级功能，依赖确定性物理——如果弹幕运动是纯数学的（不依赖 Unity 物理引擎），
>   那么任意时刻的状态都可以从初始参数重新计算，天然支持倒带。
>
> 这也是一个取巧点：**如果弹幕运动完全由数学公式驱动（而非物理模拟），
> 那么预览系统可以瞬间跳转到任意时间点，不需要逐帧模拟。**

### 缺漏6（重要）：错误检测与约束验证

编辑器需要能检测设计者的错误，例如：
- 弹幕发射点在场景边界外（弹幕一出生就被回收）
- 小怪的移动路径穿越场景边界
- 符卡时间上限设为 0 或负数
- 两个波次的时间重叠导致小怪数量超出性能预算
- 2D 模式下使用了只在 3D 中有意义的弹幕模式，但没有设置转化
- 时间轴上存在"死区"——没有任何事件发生的长时间空白

> `[🤖建议]` 编辑器应该有一个持续运行的 **验证系统**（类似 IDE 的实时错误检查）：
> - 红色：致命错误（会导致运行时崩溃或逻辑错误）
> - 黄色：警告（可能不是设计者的意图）
> - 蓝色：提示（优化建议）
>
> 验证结果显示在一个专门的面板中，点击可以跳转到对应的编辑位置。

### 缺漏7（中等）：多关卡间的共享资源管理

当前设计以"单关"为最大单位，但一个完整的游戏有多个关卡。关卡之间会共享：
- Boss 角色（同一个 Boss 可能在不同关卡以不同符卡出现）
- 小怪类型（通用杂兵在多关复用）
- 弹幕模式模板
- 场景资源（背景、特效）

> `[🤖建议]` 数据模型需要区分"项目级资源"和"关卡级资源"：
> ```
> 项目根目录/
> ├── shared/              # 项目级共享资源
> │   ├── enemies/         # 小怪类型定义
> │   ├── bosses/          # Boss 定义
> │   ├── patterns/        # 弹幕模式模板
> │   └── player.yaml      # 玩家参数
> ├── stages/              # 关卡级资源
> │   ├── stage1/
> │   │   ├── stage1.yaml  # 关卡定义
> │   │   └── assets/      # 关卡专用资源
> │   └── stage2/
> └── project.yaml         # 项目全局配置
> ```

### 缺漏8（中等）：编辑器的协作与版本控制友好性

如果有多人协作设计关卡（哪怕只是"设计者 + 程序员"两人），YAML 文件的合并冲突是不可避免的。

> `[🤖建议]` 设计数据结构时应该考虑 diff/merge 友好性：
> - 每个弹幕模式、小怪类型、符卡定义存为独立的 YAML 文件（而非一个巨大的关卡文件）
> - 使用稳定的 ID 引用（UUID 或语义化 ID），而非数组索引
> - 避免在 YAML 中存储计算得出的值（这些值会因为不同人的编辑器版本不同而产生无意义的 diff）

### 缺漏9（中等）：音频系统的深度集成

文档中提到了"BGM 波形显示"和"配合 BGM 进行情绪调动"，但音频系统的集成远不止波形显示：

- BGM 的节拍/小节标记（BPM + 拍号）——弹幕发射节奏对齐到音乐节拍
- BGM 的段落标记（前奏/主歌/副歌/间奏）——关卡段落对齐到音乐段落
- 音效触发点——符卡宣言音效、弹幕发射音效、击破音效
- 音量包络——在符卡间隔降低 BGM 音量以突出对话

> `[🤖建议]` 建议在时间轴编辑器中引入 **音乐标记轨道**：
> - 导入 BGM 后，自动或手动标记节拍点（beat markers）
> - 提供"吸附到节拍"功能——拖拽弹幕波次时自动对齐到最近的节拍
> - 这对于"弹幕配合音乐节奏"的设计体验提升是巨大的
>
> 实现上可以参考 FMOD/Wwise 的 marker 系统，或者简单地用 BPM + offset 计算。

### 缺漏10（低但影响体验）：编辑器自身的用户引导

你提到编辑器是"用户友好"的、"类似游戏的概念"。但当前设计只关注了功能，没有考虑：
- 新用户如何上手？（教程？引导流程？）
- 编辑器内是否有内置文档/提示？
- 复杂操作是否有快捷键？快捷键是否可自定义？
- 是否有"模板关卡"供新用户参考？

> `[🤖建议]` 可以在 Phase 2（弹幕编辑器）完成后就提供一个"示例弹幕库"，
> 包含经典 STG 弹幕模式的预设（自机狙、扇形弹、回转弹、随机弹等），
> 既是教学素材，也是设计者的起点。

## 十、可以取巧的设计

> `[🤖建议]` 以下是可以显著降低开发成本或提升效率的取巧策略。

### 取巧1：弹幕运动用纯数学，不用物理引擎

这是最重要的一个取巧点。

如果弹幕的运动完全由数学公式描述（位置 = f(t, 初始参数)），而不是每帧施加力/速度让物理引擎模拟，那么：

- **预览可以瞬间跳转到任意时刻**：不需要从头逐帧模拟，直接算 f(t) 就行
- **倒带免费**：t 往回减就行
- **确定性保证**：同样的参数永远产生同样的弹幕轨迹，不受帧率影响
- **序列化极简**：只需要存储公式参数，不需要存储运行时状态
- **2D⟷3D 转化简化**：转化的是公式参数，而非运行时行为

```
// 示例：一颗弹幕的位置完全由参数决定
Vector3 BulletPosition(float t, BulletParams p) {
    float r = p.speed * t;
    float theta = p.initialAngle + p.angularSpeed * t;
    float phi = p.elevationAngle + p.elevationSpeed * t;
    return p.origin + SphericalToCartesian(r, theta, phi);
}
```

代价是：弹幕不能有"碰到墙壁反弹"这种依赖运行时状态的行为——但这可以通过分段函数解决（碰壁时间可以预计算）。

### 取巧2：利用 Unity 的 AnimationCurve 做时间轴曲线编辑

Unity 内置的 `AnimationCurve` 已经有完善的曲线编辑 UI（CurveField），支持：
- 贝塞尔曲线关键帧
- 多种插值模式
- 拖拽编辑
- 序列化

与其自己实现曲线编辑器，不如直接复用 `AnimationCurve` 作为所有"随时间变化的参数"的底层类型。

```csharp
[System.Serializable]
public class CameraKeyframe {
    public AnimationCurve positionX;
    public AnimationCurve positionY;
    public AnimationCurve positionZ;
    public AnimationCurve fov;
}
```

YAML 序列化时，把 AnimationCurve 转为关键帧数组即可。

### 取巧3：ScriptableObject 做数据模型，免费获得 Undo + Inspector + 资源引用

如果把所有编辑数据都建模为 ScriptableObject：

- Unity Undo 系统直接可用（不需要自建 Command Pattern）
- Unity Inspector 直接可用（不需要从零写 UI）
- Unity 的资源引用系统直接可用（拖拽引用 Prefab、AudioClip 等）
- AssetDatabase 的搜索/过滤直接可用

代价是：ScriptableObject 是 Unity 特有的，YAML 导出需要写转换层。但这个转换层比自建 Undo 系统简单得多。

```csharp
[CreateAssetMenu(menuName = "STG/Bullet Pattern")]
public class BulletPatternAsset : ScriptableObject {
    public BulletType type;
    public int count;
    public AnimationCurve speedOverTime;
    public AnimationCurve angleOverTime;
    // Unity Undo + Inspector 自动可用
}
```

### 取巧4：时间轴 UI 复用 Unity 的 UIToolkit/IMGUI 绘制原语，不造渲染引擎

自定义时间轴编辑器最大的坑是"自己画 UI"。但 Unity 的 EditorWindow 已经提供了：
- `GUI.DrawTexture` — 画 BGM 波形
- `Handles.DrawLine` / `Handles.DrawSolidRectangleWithOutline` — 画时间轴元素
- `EditorGUILayout` — 画属性面板
- `Event.current` — 处理鼠标拖拽、键盘快捷键

不需要引入额外的 UI 框架。用 IMGUI 画时间轴虽然代码不够"优雅"，但开发速度快，且与 Unity Editor 的其他部分风格一致。

> 如果后续觉得 IMGUI 不够用，可以在 EditorWindow 内嵌 UIToolkit（VisualElement），
> 两者可以混用。

### 取巧5：弹幕模式用"发射器 + 修饰器"的组合模式，而非每种弹幕单独实现

与其为每种弹幕模式写一个类，不如设计一个组合系统：

```
发射器(Emitter)：决定弹幕从哪里、以什么空间分布发射
  ├── PointEmitter（单点）
  ├── RingEmitter（环形）
  ├── SphereEmitter（球面）
  └── LineEmitter（线形）

修饰器(Modifier)：叠加在发射器上，修改弹幕行为
  ├── SpeedCurveModifier（速度随时间变化）
  ├── HomingModifier（追踪玩家）
  ├── WaveModifier（正弦波动）
  ├── SplitModifier（飞行一段后分裂）
  └── BounceModifier（碰壁反弹）
```

一个弹幕模式 = 一个发射器 + N 个修饰器的组合。

这样做的好处：
- 新弹幕类型 = 新的修饰器组合，不需要写新代码
- 编辑器 UI 就是"选发射器 + 叠修饰器"，非常直观
- 2D⟷3D 转化只需要为每种发射器和修饰器分别定义转化规则
- 序列化天然简洁

```yaml
pattern:
  emitter:
    type: "sphere"
    count: 48
    radius: 0.5
  modifiers:
    - type: "speed_curve"
      curve: [[0, 5.0], [1, 2.0], [2, 8.0]]  # 先快后慢再加速
    - type: "wave"
      axis: "perpendicular"
      amplitude: 0.3
      frequency: 2.0
```

### 取巧6：2D⟷3D 转化——先做"投影预览"，再做真正的转化

真正的难度等价转化很复杂，但有一个几乎免费的替代方案可以先用起来：

**直接把 3D 弹幕用正交相机投影到 2D 平面上看看效果。**

这不是"正确的"2D 转化，但它能让设计者立刻看到"这个 3D 弹幕在 2D 视角下大概长什么样"。很多时候，这个投影预览就足够指导设计了。

实现成本：一个正交相机 + 一个 RenderTexture，几行代码。

真正的参数转化可以后续再做，但投影预览在 Phase 2 就能提供。

### 取巧7：符卡间隔的演出——复用镜头脚本系统做"过场动画"

符卡间隔的演出（对话、镜头运动、场景变化）和游戏内的演出脚本本质上是同一套系统。不需要为"符卡间隔"单独做一套演出工具——直接复用工具组2的场景演出脚本系统。

符卡间隔 = 一个特殊的时间段，其中：
- 弹幕系统暂停
- 玩家输入可能被限制
- 演出脚本系统接管

这样工具组2和工具组3之间就有了天然的复用关系。

### 取巧8：利用 Unity 的 Prefab Variant 做"小怪类型 + 实例"的模板系统

之前建议的"小怪类型模板 + 小怪实例"的两层结构，其实就是 Unity Prefab + Prefab Variant 的原生能力：

- 小怪类型 = Prefab（定义外观、血量、基础行为）
- 小怪实例 = Prefab Variant 或 Prefab Override（在特定关卡中的位置、路径、参数调整）

不需要自建模板系统，直接用 Unity 的。

### 取巧9：时间轴的"段落"直接对应 Unity 的 Scene

如果每个段落（道中A、中Boss战、道中B、最终Boss战）对应一个 Unity Scene：

- 场景编辑直接用 Unity Scene View
- 段落切换 = Scene 加载（Unity 原生支持 Additive Scene Loading）
- 每个段落的资源独立管理，不会互相干扰
- 编辑器中可以单独打开一个段落进行编辑

全局时间轴只需要管理"段落的排列顺序和触发条件"，不需要管段落内部的场景数据。

### 取巧10：难度指标计算——用蒙特卡洛模拟代替解析计算

之前提到的难度指标（弹幕密度、安全区域比例、反应时间等）如果要精确计算，数学会很复杂。

取巧方案：在预览环境中跑一个简单的蒙特卡洛模拟：
- 在活动区域内随机撒点
- 检测每个点在每个时刻是否会被弹幕击中
- 统计"安全点比例随时间的变化曲线"

这个方法：
- 不需要推导解析公式
- 对任意形状的弹幕模式都适用
- 精度可以通过增加采样点数来提升
- 实现简单（几十行代码）

结果可以可视化为热力图叠加在场景上——红色区域危险，绿色区域安全。
